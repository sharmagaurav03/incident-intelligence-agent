**File:** `dtest.py (or the base test class / shared fixture file that manages the connection array)`

**Problem:** CQL connections stored in the shared connections array are never closed in the majority of tests; only 2 tests call cleanup. The cassandra-driver's reconnection loop keeps the process alive for up to 2 hours after the cluster is destroyed.

**Suggested patch (illustrative, needs human review):**

```
In the base test class teardown method (e.g., tearDown or a pytest fixture with 'yield'), add a finally block that iterates over every stored connection/cluster and shuts it down:

```python
def tearDown(self):
    try:
        super().tearDown()
    finally:
        for session in getattr(self, 'connections', []):
            try:
                cluster = session.cluster
                session.shutdown()
                cluster.shutdown()
            except Exception:
                pass
        self.connections = []
```

This ensures cleanup is unconditional and applies to all tests inheriting the base class, not just the 2 that currently handle it.
```

**Diagnosis narrative:** Python DTest creates CQL connections and stores them in an array, but the teardown/cleanup logic only closes connections in 2 of the tests. The remaining tests leave connections open, causing the cassandra.pool driver to continue attempting reconnects to a now-defunct cluster for up to 2 hours (the default driver timeout/retry window), blocking process exit long after the test suite has logically completed.
