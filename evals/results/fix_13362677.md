# Fix suggestion for 13362677

**File:** `src/java/org/apache/cassandra/tools/nodetool/BootstrapMonitor.java`

**Problem:** BootstrapMonitor does not track ProgressEventType.ERROR events; after receiving COMPLETE it always exits successfully, masking prior errors.

**Suggested patch (illustrative, needs human review):**

```
--- a/src/java/org/apache/cassandra/tools/nodetool/BootstrapMonitor.java
+++ b/src/java/org/apache/cassandra/tools/nodetool/BootstrapMonitor.java
@@ -XX,XX @@
  public class BootstrapMonitor implements ProgressListener
  {
+     private volatile boolean hasError = false;
 
      @Override
      public void onEvent(ProgressEvent event)
      {
          switch (event.getType())
          {
+             case ERROR:
+                 hasError = true;
+                 System.err.println("Bootstrap error: " + event.getMessage());
+                 break;
              case COMPLETE:
+                 if (hasError)
+                     throw new RuntimeException("Bootstrap failed with errors; see messages above.");
                  // existing completion handling
                  break;
          }
      }
  }
```

**Diagnosis narrative:** The incident describes a logic bug in the `nodetool bootstrap resume` command where `BootstrapMonitor` silently ignores `ProgressEventType.ERROR` events that arrive before the `COMPLETE` event. As a result, the tool exits with a success (zero) return code even when the bootstrap operation has actually failed. The same flaw surfaces when handling failed connections. The fix requires `BootstrapMonitor` to track whether an error event was observed and propagate a non-zero exit code (or throw an exception) after the operation signals completion.
