# Fix suggestion for 13598479

**File:** `src/java/org/apache/cassandra/db/commitlog/AbstractCommitLogService.java`

**Problem:** `lastSyncedAt` is initialized with `System.currentTimeMillis()` but compared and updated exclusively with `clock.now()` (`System.nanoTime()`). The two time sources have different units (ms vs ns) and different epochs, making the comparison `lastSyncedAt + syncIntervalNanos <= pollStarted` incorrect.

**Suggested patch (illustrative, needs human review):**

```
- protected volatile long lastSyncedAt = currentTimeMillis();
+ protected volatile long lastSyncedAt = clock.now();
```

**Diagnosis narrative:** The root cause is a unit mismatch bug in AbstractCommitLogService. The field `lastSyncedAt` is initialized using `System.currentTimeMillis()` (epoch milliseconds, typically ~1.7×10¹² ms since 1970) but is subsequently read and written using `clock.now()` which wraps `System.nanoTime()` (nanoseconds since JVM start, typically a much smaller or arbitrarily-signed long). The comparison `lastSyncedAt + syncIntervalNanos <= pollStarted` is therefore semantically undefined: the two operands have incompatible origins and units. The current JDK behavior (where nanoTime values happen to be large positive longs greater than currentTimeMillis values) masks the bug, but this is not guaranteed by the JVM specification. On JVMs where nanoTime starts near zero or is negative, `lastSyncedAt` would always be greater than `pollStarted`, meaning `flushToDisk` could never become true via the time condition, stalling commit-log syncing indefinitely until a shutdown or manual sync request.
